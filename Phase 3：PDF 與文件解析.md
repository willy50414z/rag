# Phase 3：PDF 與文件解析

## 這一階段在學什麼

Phase 3 的核心不是「把 PDF 讀出來」而已，而是理解：

- PDF 為什麼常常很難處理
- 文件裡的文字、表格、圖片、版面其實是不同問題
- OCR 不是萬能，PDF parsing 也不是一定準
- 怎麼把文件內容轉成後續 RAG / extraction 能用的資料

如果 Phase 2 是學會怎麼正確使用模型，那 Phase 3 就是在學：

> 怎麼把文件變成模型真的能理解、能檢索、能抽取的輸入。

---

## 這一階段完成後，你應該會什麼

- 能分辨 text PDF 和 scanned PDF
- 知道 OCR、PDF parsing、table extraction 各自在解什麼問題
- 知道為什麼「抽出文字」不等於「理解文件」
- 能設計基本的文件轉換流程
- 能把 PDF 轉成適合 chunking、RAG、抽取的中間格式

---

## 一、為什麼 PDF 很麻煩

很多人一開始會以為 PDF 就只是文字檔的另一種格式，但實際上不是。

PDF 比較像是「排版結果」，不是「語意結構良好的文件格式」。

這代表：

- 你看到的是標題、段落、表格、頁首頁尾
- 但程式讀到的可能只是分散的文字座標
- 文件看起來整齊，不代表內部結構也整齊

所以 PDF 處理的難點通常不是「有沒有文字」，而是：

- 文字順序對不對
- 表格能不能還原
- 哪些是正文，哪些是頁首頁尾
- 哪些內容應該放在一起
- 哪些資訊需要頁碼、位置、版面輔助理解

---

## 二、PDF 處理在整個 RAG 流程中的位置

你可以把整體流程想成：

1. 取得文件
2. 判斷文件類型
3. 抽出文字 / 表格 / 版面資訊
4. 必要時做 OCR
5. 轉成中間格式
6. chunking
7. indexing
8. retrieval / extraction

Phase 3 主要在處理前半段。

如果這一段做不好，後面常見的問題會是：

- chunk 切得很怪
- RAG 找不到答案
- 抽取欄位時常常抓錯
- evidence 對不上
- hallucination 風險上升

---

## 三、先理解兩種最重要的 PDF 類型

## 1. text PDF

### 是什麼

text PDF 是指 PDF 裡面本來就有可選取文字層。

你通常可以直接：

- 反白文字
- 複製貼上
- 搜尋關鍵字

### 特性

- 不一定需要 OCR
- 抽字通常比較容易
- 但文字順序、段落、表格結構還是可能亂掉

### 常見誤解

text PDF 不代表「好處理」。

它只是表示你不一定需要 OCR，不代表版面和語意結構已經乾淨。

---

## 2. scanned PDF

### 是什麼

scanned PDF 通常是掃描文件或拍照轉 PDF，本質上每頁比較像圖片。

### 特性

- 沒有真正可用的文字層，或文字層品質很差
- 要靠 OCR 才能把影像變成文字
- OCR 品質會直接影響後續 RAG / extraction

### 常見風險

- 字被辨識錯
- 表格欄位錯位
- 中英數混雜時錯誤率升高
- 印章、手寫、歪斜、低解析度都會影響結果

---

## 四、OCR 是什麼

OCR 是 Optical Character Recognition，光學文字辨識。

你可以先把它理解成：

> 把圖片裡的字，轉成可處理的文字。

---

## 五、OCR 能做什麼，不能做什麼

## 能做什麼

- 把掃描頁面轉成文字
- 提供每個文字區塊的大致位置
- 幫你把原本不能搜尋的文件變成可搜尋

## 不能保證什麼

- 不保證辨識 100% 正確
- 不保證還原正確閱讀順序
- 不保證表格結構完整
- 不保證版面語意正確

所以 OCR 很重要，但它只是文件解析流程的一部分。

---

## 六、OCR pipeline 是什麼

OCR 不是單一步驟，通常是一條流程。

常見流程會包含：

1. 圖像前處理
2. 版面區塊切分
3. 文字辨識
4. 後處理
5. 結構重建

### 1. 圖像前處理

目的通常是提升辨識品質，例如：

- 去噪
- 旋轉校正
- 二值化
- 對比增強
- 裁切邊界

### 2. 版面區塊切分

有些系統會先判斷：

- 哪裡是標題
- 哪裡是表格
- 哪裡是正文
- 哪裡是圖片

### 3. 文字辨識

把區塊裡的圖像轉成文字。

### 4. 後處理

常見會做：

- 修正常見 OCR 錯字
- 合併被切碎的段落
- 清掉重複頁首頁尾

### 5. 結構重建

把零散文字整理成更接近人能讀的文件形式。

---

## 七、PDF Parsing 是什麼

PDF parsing 是從 PDF 裡抽出各種可用資訊的過程，不只限於文字。

常見包含：

- text extraction
- coordinate extraction
- table extraction
- image extraction
- layout analysis

你可以把它理解成：

> OCR 比較像是「看圖讀字」，PDF parsing 比較像是「拆解文件內容與位置」。

---

## 八、常見的 parsing 任務

## 1. text extraction

### 是什麼

把 PDF 中的文字抽出來。

### 問題點

- 閱讀順序可能錯
- 換行可能亂
- 多欄版面可能混在一起
- 表格內容可能被攤平成破碎文字

### 你要知道的事

抽出來的文字看起來能讀，不代表適合直接進 RAG。

---

## 2. coordinate extraction

### 是什麼

除了文字本身，也保留它在頁面上的位置。

### 為什麼重要

因為很多語意其實來自位置：

- 標題通常在上方
- 欄位名稱與欄位值可能左右對應
- 表格是靠空間排列，不只是靠文字內容

### 實務用途

- 做 evidence 高亮
- 還原欄位對應
- 幫助 table extraction
- 協助 layout-aware extraction

---

## 3. table extraction

### 是什麼

把表格從 PDF 中抽出來，轉成有列、欄結構的資料。

### 為什麼困難

因為 PDF 裡的表格不一定真的有「表格物件」，有時只是排版看起來像表格。

### 常見難點

- 無框線表格
- 合併儲存格
- 跨頁表格
- 欄位對不齊
- 掃描品質差

### 重要觀念

表格如果沒正確抽出來，後面的抽取結果通常會很差。

---

## 4. image extraction

### 是什麼

把 PDF 中的圖片抽出來。

### 什麼時候重要

- 圖片裡其實有文字
- 圖表本身有資訊
- 附件是掃描圖片嵌在 PDF 裡

---

## 九、Layout 是什麼，為什麼重要

layout 指的是版面結構，例如：

- 標題
- 段落
- 欄位區塊
- 表格
- 頁首頁尾
- 多欄排版

對人來說，layout 很自然；但對系統來說，如果只拿到一串文字，很多語意會消失。

例如：

- 「總金額」在左邊，「12345」在右邊
- 如果位置資訊消失，系統不一定知道兩者是一組

所以文件解析常常不只是取文字，而是盡量保留：

- 頁碼
- block
- section
- 座標
- 表格結構

---

## 十、Chunking 在這一階段為什麼已經要開始想

很多人把 chunking 當 Phase 4 才要想的事，但其實 Phase 3 就要開始考慮。

因為你怎麼解析文件，會直接決定你後面怎麼切 chunk。

例如：

- 如果段落本來就亂掉，chunk 再怎麼切都不會漂亮
- 如果表格被拆碎，RAG 很難抓到完整答案
- 如果頁首頁尾混進正文，retrieval 品質會下降

所以 Phase 3 的重點之一是：

> 先把文件整理成適合 chunking 的中間表示。

---

## 十一、Chunking 常見名詞

## 1. semantic chunk

### 是什麼

依語意切段，而不是只靠固定字數。

### 例子

- 一個完整小節是一個 chunk
- 一段條款不要切到一半

### 為什麼重要

語意完整的 chunk 比硬切字數更有利於 retrieval 與 grounding。

---

## 2. overlap

### 是什麼

相鄰 chunk 之間保留一部分重疊內容。

### 為什麼需要

有些關鍵資訊剛好落在段落交界處，如果完全不重疊，可能被切斷。

### 注意

overlap 不是越多越好，太多會增加重複內容與成本。

---

## 3. section-aware chunking

### 是什麼

切 chunk 時參考文件結構，例如：

- 章節標題
- 小節
- 表格區塊
- 附錄

### 為什麼重要

這通常比純字數切法更接近文件實際語意。

---

## 4. metadata

### 是什麼

metadata 是描述 chunk 或文件的附加資訊。

常見包含：

- 檔名
- 頁碼
- section title
- block id
- table id
- document type

### 為什麼重要

metadata 能幫你：

- 過濾搜尋範圍
- 回答時附引用
- 做權限控制
- 追蹤錯誤來源

---

## 十二、文件中間格式是什麼

中間格式是指：

你不要直接把原始 PDF 丟給後續系統，而是先轉成一個比較穩定、可追蹤、可處理的表示。

常見形式有：

- Markdown
- JSON
- 結構化 document object

---

## 十三、為什麼常會轉成 Markdown

Markdown 的好處是：

- 對人可讀
- 對 LLM 相對友善
- 容易保留標題層級
- 容易把段落、清單、表格做初步整理

### 但要注意

Markdown 不是萬能。

如果你有很強的表格需求、欄位定位需求，通常還是要另外保留 JSON 結構與位置資訊。

---

## 十四、為什麼常會轉成 JSON

JSON 比較適合機器處理，因為它可以清楚描述：

- 頁面
- 區塊
- 段落
- 表格
- 座標
- metadata

### 一個簡化例子

```json
{
  "document_id": "doc_001",
  "pages": [
    {
      "page_number": 1,
      "blocks": [
        {
          "type": "paragraph",
          "text": "This agreement is made on ...",
          "bbox": [10, 20, 400, 80]
        }
      ]
    }
  ]
}
```

### 實務意義

Markdown 比較適合讀與餵模型，JSON 比較適合做系統處理與 traceability。

很多情況下兩者都值得保留。

---

## 十五、Phase 3 在 RAG / extraction 中真正重要的觀念

如果你是為了後面的文件問答或欄位抽取做準備，這一階段最重要的不是背工具名，而是先建立以下觀念：

1. PDF 看起來像文字，其實本質更像版面資料
2. OCR 只是把圖像轉成文字，不等於理解文件
3. 表格、段落、標題、頁首頁尾要分開思考
4. 位置資訊常常和文字本身一樣重要
5. 中間格式品質，會直接決定 RAG 與 extraction 品質

---

## 十六、初學者最常見的誤區

## 1. 以為能複製文字就代表解析完成

錯。你還要處理順序、結構、表格、位置。

## 2. 以為 OCR 完就能直接做抽取

錯。OCR 後通常還要清理、切塊、補 metadata。

## 3. 以為只要拿全文丟給 LLM 就好

錯。成本高、不穩定，而且很難做 grounding。

## 4. 以為表格只是普通文字

錯。表格通常需要單獨處理。

## 5. 以為 chunking 只是固定長度切字

錯。好的 chunking 依賴前面的文件結構整理。

---

## 十七、你應該怎麼學 Phase 3

建議順序如下：

1. 先分清楚 text PDF 和 scanned PDF
2. 理解 OCR 在做什麼、限制是什麼
3. 理解 text extraction、table extraction、coordinate extraction 的差異
4. 練習把 PDF 轉成 Markdown
5. 再把 PDF 轉成保留頁碼與 block 的 JSON
6. 最後再去思考 chunking 與 metadata 設計

---

## 十八、Phase 3 的最小實作練習

你可以做一個最小專案：

### 題目

把一份 PDF 轉成後續 RAG 可用的中間格式。

### 最低要求

- 判斷這份 PDF 是 text PDF 還是 scanned PDF
- 如果是 scanned PDF，走 OCR
- 輸出一份 Markdown
- 輸出一份 JSON
- JSON 要至少保留頁碼、區塊、文字內容

### 進階要求

- 嘗試分出標題與段落
- 清掉頁首頁尾
- 額外保留表格區塊

---

## 十九、你可以用這個標準判斷自己有沒有學會

如果你已經能做到下面幾件事，就代表 Phase 3 基本過關：

- 你能解釋 text PDF 和 scanned PDF 的差別
- 你知道 OCR 解決的是哪一段問題
- 你知道為什麼文字順序與座標資訊很重要
- 你知道 Markdown 和 JSON 各自適合什麼用途
- 你知道文件解析品質會直接影響後面的 RAG

---

## 二十、給你的簡化結論

Phase 3 可以濃縮成一句話：

> 把原本對人可讀、但對系統難用的 PDF，轉成可檢索、可抽取、可追溯的結構化文件表示。

這一階段的重點不是某個工具，而是文件表示品質。

只要你抓住下面這條主線，後面做 RAG 會順很多：

- 先判斷文件類型
- 再決定要不要 OCR
- 盡量保留結構與位置
- 轉成穩定中間格式
- 為後面的 chunking、retrieval、extraction 做準備

---

## 二十一、下一步建議

你現在最適合做的練習不是追更多 parsing 名詞，而是實際找 2 到 3 份不同 PDF 來測：

1. 一份可選字的 text PDF
2. 一份掃描型 PDF
3. 一份含表格的 PDF

對每份文件都觀察：

- 能不能直接抽字
- 文字順序是否正確
- 表格有沒有壞掉
- 頁碼與區塊能不能保留
- 哪些內容如果直接拿去做 RAG 會出事

做完這個練習，你對 Phase 3 的理解會比只看術語快很多。
