# Phase 2 附件：Hallucination Detection

## 這份附件在講什麼

在 Phase 2，你會接觸很多 LLM 名詞，但真正進入實作後，最容易踩到的問題之一就是 hallucination。

hallucination 不是單純指「模型答錯」，而是：

- 模型輸出了看似合理、實際上沒有根據的內容
- 模型補出了你沒有提供的事實
- 模型把不確定的資訊說得很肯定

這份附件的目的，是幫你建立一個實務觀念：

> hallucination 通常不能只靠單一方法發現，而是要靠多層檢查。

---

## 一、先講結論

hallucination 一般不是只能透過 `validation` 或人工審閱發現，但這兩者確實很重要。

更完整的做法通常是這樣：

1. 先要求模型只根據提供內容回答
2. 要求模型附上證據或引用來源
3. 對輸出做 schema validation
4. 對結果做 business rule 檢查
5. 檢查答案是否真的被原文支持
6. 低信心或高風險案例送人工審閱

所以問題的重點不是「是不是只能靠 validation 或人工」，而是：

> 你要怎麼把 hallucination 變成一個可觀察、可攔截、可分流的工程問題。

---

## 二、什麼情況最容易 hallucination

以下情況特別常見：

- 使用者問題很模糊，但模型還是硬答
- 原始資料不完整，模型自己補空白
- prompt 沒有明確要求「不知道就說不知道」
- 你要求自由回答，沒有要求輸出結構與依據
- RAG 檢索回來的內容不夠，但模型還是往下生成
- 抽取欄位時，文件沒有值，模型卻猜了一個值

---

## 三、hallucination 怎麼發現

## 1. Grounding 檢查

### 核心問題

模型說的內容，能不能在原始資料裡找到依據。

### 例子

如果模型回答：

- 合約終止日是 2025-12-31

但你回頭查文件，根本沒有這個日期，這就是高風險 hallucination。

### 實務做法

- 要模型回答時附上引用句子
- 保留頁碼、段落、chunk id
- 對照答案與原始文本是否一致

### 這種方法能抓什麼

- 編造來源
- 補出文件沒有的欄位
- 把不在文件中的內容說成事實

---

## 2. Retrieval 對照

### 核心問題

在 RAG 裡，回答是否真的被取回來的 chunk 支持。

### 常見情況

模型回答得很完整，但你一看 retrieval 結果，chunk 裡根本沒有那些內容。

這通常代表兩種可能：

- retrieval 沒抓到正確資料
- 模型超出證據亂補

### 實務做法

- 把最終答案和 top-k chunk 一起保存
- 檢查答案中的關鍵事實是否能在 chunk 中找到
- 如果找不到，標記為 unsupported answer

### 這種方法能抓什麼

- RAG 中最典型的「有回答但沒根據」

---

## 3. Schema Validation

### 核心問題

輸出格式是否合法、欄位是否完整、型別是否正確。

### 可以抓到的問題

- JSON 格式壞掉
- 欄位缺失
- 型別錯誤
- enum 不合法

### 抓不到的問題

schema validation 抓不到這種情況：

- 格式正確
- 欄位完整
- 但內容其實是模型編的

### 結論

validation 很重要，但它不是 hallucination 偵測的全部。

---

## 4. Rule-based 檢查

### 核心問題

結果是否違反明確規則或業務邏輯。

### 例子

- 金額不能小於 0
- 稅前金額 + 稅額應等於總金額
- 結束日期不能早於開始日期
- 出生日期不應晚於今天

### 為什麼有效

模型很常生成「看起來像真的」數值，但這些數值可能不符合邏輯。

### 這種方法能抓什麼

- 邏輯型 hallucination
- 數值型 hallucination
- 關聯欄位彼此矛盾

---

## 5. Ask for Evidence

### 核心問題

不要只要答案，還要答案的依據。

### 可以要求模型一起輸出

- 原文引用
- 頁碼
- chunk id
- 支持該欄位的句子
- confidence

### 例子

不要只輸出：

```json
{
  "invoice_date": "2024-05-03"
}
```

可以改成：

```json
{
  "invoice_date": "2024-05-03",
  "evidence": "Invoice Date: 2024-05-03",
  "page": 1
}
```

### 為什麼重要

這會讓 hallucination 更容易被發現，因為你不再只看結果，而是看結果能不能對回原文。

---

## 6. Cross-check / Secondary Verification

### 核心問題

讓另一個步驟來驗證第一個步驟的輸出。

### 例子

第一步：

- 模型先抽取欄位

第二步：

- 再讓另一個 prompt 檢查「這些欄位是否都能從原文支持」

### 實務理解

這不代表第二次模型一定更準，而是把流程拆成：

- 產生答案
- 檢查答案

這比只做一次生成更容易觀察風險。

---

## 7. Uncertainty Signals

### 核心問題

觀察哪些輸出行為很像 hallucination。

### 常見危險訊號

- 回答過度完整
- 對模糊問題過度肯定
- 補出原文沒有的細節
- 多跑幾次結果差很多
- confidence 很高，但 evidence 很弱
- 模型使用了「推測」、「應該」、「可能」卻仍輸出確定值

### 用途

這些訊號不一定等於 hallucination，但適合用來做風險分流。

---

## 8. Adversarial Testing

### 核心問題

故意拿模糊、缺漏、矛盾資料去測模型會不會亂猜。

### 測試方式

- 故意移除重要欄位
- 故意放兩個互相矛盾的日期
- 故意讓金額不一致
- 故意提供空白頁或 OCR 品質差的內容

### 為什麼有效

如果模型在這些情況下還是很肯定地輸出答案，通常代表你的 prompt 或驗證機制還不夠。

---

## 9. Human Review

### 核心問題

當風險太高、資料太重要，不能只靠自動化判斷。

### 適合人工審閱的情況

- 財務資料
- 法務文件
- 醫療資訊
- 要正式入庫的關鍵欄位
- evidence 對不上原文
- confidence 太低

### 正確理解

人工審閱不是系統失敗，而是風險控制的一部分。

---

## 四、不同方法能抓到什麼

| 方法 | 能抓到什麼 | 抓不到什麼 |
| --- | --- | --- |
| Grounding 檢查 | 答案無法對回原文 | 原文本身就有錯的情況 |
| Retrieval 對照 | RAG 答案沒被 chunk 支持 | chunk 雖有內容，但理解錯誤 |
| Schema validation | 格式錯、型別錯、缺欄位 | 格式正確但內容是編的 |
| Rule-based 檢查 | 邏輯衝突、數值不合理 | 看似合理但來源錯誤 |
| Ask for evidence | 可追溯性、來源對照 | evidence 本身被抽錯但看起來像真的 |
| Cross-check | 流程性驗證、互相校對 | 兩次都一起錯 |
| Human review | 高風險最終把關 | 成本高、不可完全擴張 |

---

## 五、在 RAG 專案裡最實用的做法

如果你是在做文件問答或結構化抽取，最實際的組合通常是：

1. prompt 明確要求「只能根據提供內容回答」
2. 找不到就輸出 `null` 或 `not found`
3. 輸出固定 JSON
4. 每個欄位盡量附 evidence
5. 做 schema validation
6. 做 business rule 檢查
7. evidence 不足或 confidence 太低時，送 review queue

這套做法比只靠「模型再想一次」穩很多。

---

## 六、實務上很重要的一句話

> hallucination 不是只看答案對不對，而是看答案有沒有根據。

這個差異很重要。

因為有些答案剛好答對了，但它其實是猜的。這種情況在系統設計上仍然是風險。

---

## 七、你在 Phase 2 應該學到什麼程度

這份附件不是要你現在就做完整的 hallucination defense system，而是要你先建立正確觀念：

- LLM 可能會亂補
- 格式正確不代表內容可信
- 沒有 evidence 的答案風險很高
- validation 只能解決一部分問題
- RAG 最關鍵的是 grounding，不是回答看起來多自然

如果你已經理解這幾點，Phase 2 的基礎方向就對了。

---

## 八、給你的簡化版檢查清單

每次你用 LLM 做抽取、問答、分類時，都可以快速問自己：

1. 這個答案有沒有明確來源？
2. 這個欄位是抽出來的，還是猜出來的？
3. 如果資料不存在，模型會不會硬補？
4. 結果是否通過 schema validation？
5. 結果是否違反業務規則？
6. 低信心案例有沒有人工覆核入口？

---

## 九、最後結論

hallucination 的發現方式，不是只靠 validation 或人工審閱。

更成熟的做法是把它拆成多層：

- 來源對照
- retrieval 對照
- 結構驗證
- 規則檢查
- 證據要求
- 二次驗證
- 人工覆核

對 RAG 來說，最核心的一條原則是：

> 沒有被證據支持的答案，就不應該輕易信任。
